⚡ 微任务插队机制（面试必考）

核心规则：每个宏任务阶段执行完毕后、切换到下一阶段之前，Node 会先清空微任务队列。顺序严格为：
① 清空 nextTick 队列（全部）→ ② 清空 Promise 微任务队列（全部）

🧪 经典面试题：逐行执行跟踪

// 完整执行顺序是什么？
console.log('1. sync start');

setTimeout(() => console.log('2. setTimeout'), 0);

setImmediate(() => console.log('3. setImmediate'));

Promise.resolve()
  .then(() => console.log('4. Promise.then 1'))
  .then(() => console.log('5. Promise.then 2'));

process.nextTick(() => console.log('6. nextTick 1'));
process.nextTick(() => {
  console.log('7. nextTick 2');
  process.nextTick(() => console.log('8. nextTick 3 (嵌套)'));
});

console.log('9. sync end');
📋 执行顺序 + 详解：
=== 同步代码（调用栈） === 1. sync start ← setTimeout → timers / setImmediate → check 9. sync end ← Promise / nextTick → 微任务队列 === 同步栈清空 → 微任务检查 === 6. nextTick 1 ← nextTick 优先于 Promise！ 7. nextTick 2 ← 执行过程中追加了 8 8. nextTick 3 (嵌套) ← nextTick 队列必须全部清空 4. Promise.then 1 ← nextTick 清空后 → Promise 5. Promise.then 2 ← .then 链入队 === 微任务清空 → Event Loop === → ① timers：setTimeout 到期 2. setTimeout → ②③④ 流转... → ⑤ check 3. setImmediate

⭐ 关键结论（背下来）

  1. nextTick > Promise.then：每次微任务检查，先清空 nextTick 再清空 Promise
  2. setImmediate > setTimeout(0)：check 在 poll 之后立刻执行，setTimeout 等下轮 timers
    例外：在 I/O 回调内部，setImmediate 永远先于 setTimeout——因为 I/O 回调在 poll 执行，poll 之后就是 check
  3. nextTick 嵌套不降级：回调里再调 nextTick，追加到当前队列，本轮清空（可能"饿死"事件循环）
  4. Promise.then 链式调用：每个 .then 是一次新的微任务入队，级联执行

setTimeout(fn, 0) vs setImmediate(fn) —— 面试送命题

这两者最常见的误区是"谁先执行"。答案：看调用位置。

// 在主模块顶层 —— 顺序不确定
setTimeout(() => console.log('timeout'), 0);
setImmediate(() => console.log('immediate'));

// 实际输出随机：timeout → immediate 或 immediate → timeout
// 原因：setTimeout(fn, 0) 被强制截断为 setTimeout(fn, 1)
//       如果 Event Loop 启动耗时 > 1ms，则 setTimeout 先到期 → timeout 先执行
//       如果 Event Loop 启动耗时 < 1ms，则 poll 阶段跳过 timers → immediate 先执行
// 在 I/O 回调内部 —— setImmediate 永远先执行（确定性结果）
fs.readFile('/tmp/test.txt', () => {
  setTimeout(() => console.log('timeout'), 0);
  setImmediate(() => console.log('immediate'));
});

// 输出一定是：immediate → timeout
// 原因：I/O 回调在 poll 阶段执行 → poll 之后立刻进入 check → setImmediate 先
//        等下轮循环的 timers 阶段，setTimeout 才执行
🎯 面试一句话：在 I/O 回调里，setImmediate 永远先于 setTimeout。在主模块顶层，顺序不确定——取决于 Event Loop 启动耗时。

setInterval 的两个坑

坑一：不精确，会漂移。setInterval 的间隔是"上次回调开始之后"算，不是"上次回调结束之后"。如果回调执行了 50ms，那间隔就短了 50ms。

let last = Date.now();
setInterval(() => {
  const now = Date.now();
  console.log('实际间隔', now - last, 'ms');
  last = now;
  // 模拟耗时操作
  let x = 0; for (let i = 0; i < 1e8; i++) x++;
}, 100);

// 输出：实际间隔 180ms、220ms...（间隔为 100 但被回调执行时间挤占，然后补偿）
// 也可能：实际间隔 50ms、70ms...（上次回调霸占了时间槽，下次被立即触发）

坑二：无视回调是否完成就堆叠。如果回调执行时间 > 间隔，新的回调会排队堆积（Node 会跳过已排队的回调，但不会取消还未排队的）。

更好的替代方案：用 setTimeout 递归模拟精确间隔：

function preciseInterval(fn, delay) {
  let running = true;
  function tick() {
    if (!running) return;
    const start = Date.now();
    fn();
    const elapsed = Date.now() - start;
    setTimeout(tick, Math.max(0, delay - elapsed));  // 校准：扣掉执行时间
  }
  setTimeout(tick, delay);
  return () => { running = false; };
}

1.2 闭包 / 作用域详解

🏠 一个比喻：函数的"老家"

每个函数都有一个"老家"——就是它被定义时所在的那个代码块。不管这个函数后来被扔到哪儿去调用，它永远记得老家里的变量。

这就好比一个人出生在上海，长大后去了北京工作——他依然知道自己上海老家有哪些东西（爸妈、房子、儿时玩具）。这就是词法作用域的核心：函数能访问什么变量，只取决于它在哪儿被定义的，不取决于在哪儿被调用。

一句话总结：闭包 = 函数 + 它能记住的周围的变量。闭包不是 JS 故意设计的特性，而是词法作用域的自然结果——只要你把函数定义在另一个函数里面，闭包就自动产生了。

🔗 看一个最简单的例子

function makeGreeter(name) {
  const greeting = '你好，';
  // ↓ innerGreet 定义在这里，"老家"是 makeGreeter 内部
  return function innerGreet() {
    console.log(greeting + name);  // 用到了"老家"的 greeting 和 name
  };
}

const greetXiaoMing = makeGreeter('小明');
// makeGreeter 已经执行完了，按理说 greeting 和 name 应该销毁了...
greetXiaoMing();  // "你好，小明"  ← 但 innerGreet 还"记得"它们！

为什么 makeGreeter 都跑完了，greeting 和 name 还在？因为 innerGreet 把老家带在身上——这就是闭包。

📦 闭包最常见的用途：创建私有变量

JS 没有 private 关键字的时候，全靠闭包实现：

function createCounter() {
  let count = 0;  // 这个变量外面摸不到
  return {
    increment: () => ++count,   // 只能通过这两个方法操作
    getCount:  () => count,
  };
}

const counter = createCounter();
counter.increment();
counter.increment();
console.log(counter.getCount());  // 2
// console.log(counter.count);   // undefined ← 外面看不到 count

🔍 变量查找：从里往外找，就近原则

当你在函数里用到一个变量，JS 引擎怎么找到它？很简单：

const globalVar = '我是全局';

function outer() {
  const outerVar = '我是外层';

  function inner() {
    const innerVar = '我是内层';
    console.log(innerVar);   // ① 先在自己这层找 → 找到了
    console.log(outerVar);   // ② 自己没找到 → 往上一层找 → 找到了
    console.log(globalVar);  // ③ 再往上 → 找到了
    console.log(nonExist);   // ④ 找到最外层都没有 → ReferenceError 💥
  }

  inner();
}

outer();

这个从里往外一层层找的链条就是作用域链。规则很朴素：就近覆盖，内层优先——如果里外有同名变量，里层的会"遮蔽"外层的。

⚠️ 关键认知：闭包查找变量走的是定义时的层级关系（源代码里谁套在谁里面），不是运行时的调用关系。B 函数调用了 A 函数，不代表 A 能访问 B 的变量——A 只看自己的"老家"。

💀 闭包的内存问题：什么时候会翻车

闭包本身不会泄漏内存。问题出在闭包活得太久，还抓着一些它不需要的大东西。

❌ 场景一：无心抓了大对象

function processLargeData(data) {
  const hugeBuffer = Buffer.alloc(100 * 1024 * 1024); // 100MB

  const getFirstLine = () => data.split('\n')[0];  // 只需要 data

  // ❌ getFirstLine 和 hugeBuffer 在同一个"老家"
  //    即使 getFirstLine 根本没用到 hugeBuffer，GC 也可能不敢回收它

  return getFirstLine;
}

✅ 修复：把大对象放到闭包定义之后

function processLargeData(data) {
  const getFirstLine = () => data.split('\n')[0];  // 先定义闭包

  // ✅ hugeBuffer 在后面，不在 getFirstLine 的"老家"里
  const hugeBuffer = Buffer.alloc(100 * 1024 * 1024);
  doSomething(hugeBuffer);

  return getFirstLine;
}

❌ 场景二：事件监听忘了关

class Connection {
  constructor(socket) {
    this.socket = socket;
    this.handleData = (data) => {           // 箭头函数 = 闭包
      this.process(data);                    // 持有 this → Connection 实例
    };
    socket.on('data', this.handleData);
  }
  // ❌ socket 上的监听器持有 handleData → handleData 持有 this
  //    → 只要 socket 还活着，Connection 实例就永远无法被 GC
}

✅ 修复：提供清理方法，解绑再丢弃

class Connection {
  constructor(socket) {
    this.socket = socket;
    this.handleData = (data) => this.process(data);
    socket.on('data', this.handleData);
  }
  destroy() {
    this.socket.off('data', this.handleData);  // ✅ 断开引用链
    this.socket = null;
  }
}

❌ 场景三：定时器里的闭包——你以为丢了，其实还在

let server = new HeavyServer();

setInterval(() => {
  console.log(server.getStatus());  // 闭包捕获了 server
}, 1000);

// 你"以为"这样可以释放：
server = null;  // ❌ 无效！定时器里的箭头函数还抓着原来的 server
✅ 正确做法：先 clearInterval(id)，再 server = null。定时器本身不死，谁也收不走里面的变量。

✅ 三条实用守则

  1. 闭包尽量"小"：把闭包定义在刚好需要那些变量的位置，不要随手写在函数最外层
  2. 用完要"断"：事件监听 off、定时器 clearInterval、setTimeout 用完就 clearTimeout
  3. 全局的闭包 ≈ 永远不死的闭包：全局变量、模块级变量上的闭包几乎等于永久持有引用，写的时候多留个心眼
💡 记一句话就够了：闭包是"函数出生证明"——在哪儿出生，就拥有哪儿的变量访问权，终身有效。写闭包时问自己：它要活多久？它抓着的东西有多大？

1.3 原型链 / 类详解

JS 的继承不走传统的"类继承"路线，而是原型继承——对象直接链到另一个对象，找不到的属性沿着链往上找。没有类拷贝，只有链式查找。

理解原型链只需要搞清两个东西：prototype（函数上的共享仓库）和 __proto__（每个对象上的查找指针）。别被名字迷惑，看代码就清楚了：

👁️ 直接看代码，比文字好理解

function Person(name) {
  this.name = name;               // 每个实例自己的属性
}

// 挂在 Person.prototype 上的方法，所有实例共享
Person.prototype.sayHi = function() {
  console.log('你好，我是' + this.name);
};

const alice = new Person('Alice');
const bob   = new Person('Bob');

// 关键关系：
console.log(alice.__proto__ === Person.prototype);  // true ← 核心等式！
// alice 的"备用查找源" 就是 Person 的"共享方法仓库"

// 属性查找：
console.log(alice.name);    // 'Alice'  ← 自己就有，直接用
console.log(alice.sayHi);   // function ← 自己没有，沿着 __proto__ 找到 Person.prototype.sayHi

// 方法共享验证：
console.log(alice.sayHi === bob.sayHi);  // true ← 同一个函数，不是每个实例复制一份
📌 记住这个等式就够了：实例.__proto__ === 构造函数.prototype。理解了这一句，原型链就通了一半。

⛓️ 原型链：找不到就一直往上找

JS 的属性查找就是沿着 __proto__ 一层层往上爬，直到找到或者爬到顶（null）。

// 延续上面的 Person 例子
console.log(alice.__proto__);                    // Person.prototype
console.log(alice.__proto__.__proto__);           // Object.prototype
console.log(alice.__proto__.__proto__.__proto__); // null ← 到头了

// 所以 alice 可以用 hasOwnProperty——它不在 Person.prototype 上，在 Object.prototype 上
console.log(alice.hasOwnProperty('name'));  // true ← 沿着原型链在 Object.prototype 上找到的
原型链终点：所有普通对象的原型链最终都通向 Object.prototype，然后 Object.prototype.__proto__ === null。null 就是这条链的终点——找不到就返回 undefined。

🔄 ES6 class = 原型链的语法糖

class 关键字没有创造新的继承机制，它只是把上面的 function + prototype 写法包装得更像 Java/C++。底层一模一样。

class 写法 vs 它背后的真相

// ===== class 写法（你写的）=====
class Animal {
  constructor(name) {
    this.name = name;
  }
  speak() {
    console.log(this.name + ' 叫了一声');
  }
}

class Dog extends Animal {
  constructor(name, breed) {
    super(name);      // ← 相当于 Animal.call(this, name)
    this.breed = breed;
  }
  speak() {
    console.log(this.name + ' 汪汪叫');
  }
}

const wangcai = new Dog('旺财', '柴犬');
wangcai.speak();  // 旺财 汪汪叫
// ===== 底层实际执行的事（等价的原型链写法）=====

function Animal(name) {
  this.name = name;
}
Animal.prototype.speak = function() {
  console.log(this.name + ' 叫了一声');
};

function Dog(name, breed) {
  Animal.call(this, name);   // ← super(name) 就是这个
  this.breed = breed;
}

// ↓ extends 就是这两步：
Dog.prototype = Object.create(Animal.prototype);  // ① Dog.prototype 的原型指向 Animal.prototype
Dog.prototype.constructor = Dog;                   // ② 修正 constructor 指向

Dog.prototype.speak = function() {
  console.log(this.name + ' 汪汪叫');
};
🔑 class 三个关键字的底层等价：
constructor()= 构造函数本身
method()= Constructor.prototype.method = function()
extends= Child.prototype = Object.create(Parent.prototype)
super()= Parent.call(this, ...)

🔬 instanceof：沿着原型链找人

instanceof 不关心你在哪个构造函数里出生，它只做一件事：沿着右边的 prototype 在左边的原型链上找。

const arr = [1, 2, 3];

console.log(arr instanceof Array);     // true  ← arr.__proto__ 链上有 Array.prototype
console.log(arr instanceof Object);    // true  ← arr.__proto__.__proto__ 是 Object.prototype
console.log(arr instanceof Function);  // false ← Function.prototype 不在 arr 的原型链上

🔬 getPrototypeOf vs typeof：两个容易搞混的东西

1.4 迭代器 / 生成器详解

🚶 迭代器：一个能"一步步走"的东西

迭代器的概念很简单：一个对象，每次调 .next() 就吐出下一个值，吐完了就说 done: true。就像自动贩卖机——按一下出一罐，卖完了亮红灯。

// 一个最简单的迭代器
let i = 0;
const iterator = {
  next() {
    if (i < 3) {
      return { value: i++, done: false };  // 有货，吐出 value
    }
    return { value: undefined, done: true };   // 卖完了
  }
};

console.log(iterator.next());  // { value: 0, done: false }
console.log(iterator.next());  // { value: 1, done: false }
console.log(iterator.next());  // { value: 2, done: false }
console.log(iterator.next());  // { value: undefined, done: true } ← 没了

这就是迭代器的全部：一个对象，有 next() 方法，返回 { value, done }。

🔁 Iterable（可迭代对象）：给自己挂一个"默认迭代器"

上面的迭代器有一个问题：你没法用 for...of 遍历它。要支持 for...of，对象必须实现 Symbol.iterator 方法——这个方法返回一个迭代器。

// 一个自定义的可迭代对象
const range = {
  from: 1,
  to: 5,

  // Symbol.iterator 让 range 变成"可迭代的"
  [Symbol.iterator]() {
    let current = this.from;
    const end = this.to;
    return {
      next() {
        if (current <= end) {
          return { value: current++, done: false };
        }
        return { value: undefined, done: true };
      }
    };
  }
};

// 现在可以用 for...of 了！
for (const num of range) {
  console.log(num);  // 1, 2, 3, 4, 5
}

// 也可以用展开运算符
console.log([...range]);  // [1, 2, 3, 4, 5]
console.log(Math.max(...range));  // 5
🎯 分清两个概念：
Iterator（迭代器） = 有 next() 方法的对象，返回 { value, done }
Iterable（可迭代对象） = 有 [Symbol.iterator]() 方法的对象，可以被 for...of 遍历
一个对象可以同时是两者（像内置的数组、Map、Set 一样）。

📦 内置的可迭代对象

JS 里这些东西都自带 Symbol.iterator：

const arr = ['a', 'b', 'c'];
const set = new Set([1, 2, 3]);
const map = new Map([['k', 'v']]);
const str = 'hello';

// 全都能用 for...of
for (const item of arr)  {}  // ✅ Array
for (const item of set)  {}  // ✅ Set
for (const item of map)  {}  // ✅ Map → 每次吐出 [key, value]
for (const char of str)  {}  // ✅ String → 逐字符

⚡ 生成器（Generator）：迭代器的"懒人包"

手写 next() 太啰嗦了。生成器让你用 yield 关键字直接"暂停"函数，JS 自动帮你生成 next() 方法。

// function* 定义一个生成器函数
function* simpleGenerator() {
  console.log('开始');
  yield 1;              // 暂停，吐出 1，等下次 next()
  console.log('中间');
  yield 2;              // 暂停，吐出 2
  console.log('结束');
  return 3;             // return 值也会出现在 { value: 3, done: true }
}

const gen = simpleGenerator();  // 调用生成器函数，得到生成器对象（什么代码都没执行！）

console.log(gen.next());  // "开始" → { value: 1, done: false }
console.log(gen.next());  // "中间" → { value: 2, done: false }
console.log(gen.next());  // "结束" → { value: 3, done: true }
生成器三件事：
① function* 定义的函数，调用后不执行，返回一个生成器对象
② 每次 .next() 执行到下一个 yield，暂停在那里
③ yield 后面的表达式的值就是 { value: ..., done: false } 里的 value

🔄 生成器实现的 Symbol.iterator（比手写简洁 10 倍）

// 刚才手写的 range，用生成器重写
const range = {
  from: 1,
  to: 5,

  *[Symbol.iterator]() {       // * 在方法名前 = 生成器方法（ES6 语法）
    for (let i = this.from; i <= this.to; i++) {
      yield i;                   // 每次 yield 暂停，for...of 自动调 next()
    }
  }
};

console.log([...range]);  // [1, 2, 3, 4, 5] ← 效果一样，代码少了一大半

📨 yield 还能"收"数据：双向通信

yield 不仅能往外吐值，next(arg) 还能往里传值——这是生成器的独门绝技：

function* twoWay() {
  const a = yield '给我第一个值';   // 先吐出提示，暂停，等 next(val) 把 val 赋给 a
  console.log('收到了:', a);
  const b = yield '给我第二个值';
  console.log('收到了:', b);
  return '完事';
}

const gen = twoWay();
console.log(gen.next().value);       // "给我第一个值"  ← 第一个 next() 不能传参，传了也被忽略
console.log(gen.next('hello').value); // "收到了: hello" → "给我第二个值"
console.log(gen.next('world').value); // "收到了: world" → "完事"
⚠️ 注意：第一个 next() 不能传参，因为还没有 yield 在等它。从第二个 next(val) 开始，val 会赋给上一个 yield 的返回值。

🧵 生成器做异步流程控制：让异步看起来像同步

痛点：连续发两个请求，得层层嵌套：

// 回调地狱（不好读）
fetchUser('Bob', user => {
  fetchOrders(user.id, orders => {
    console.log(orders);  // 终于拿到
  });
});

生成器 + yield 可以把它”拉直”。原理分三步：

第 1 步：用 yield 暂停，把异步操作”吐出去”

function* myFlow() {
  const user   = yield fetchUser('Bob');   // ① 停在这儿，把 fetchUser 吐出去
  const orders = yield fetchOrders(user.id); // ② 拿到 user 后继续，再停，把 fetchOrders 吐出去
  console.log(orders);                         // ③ 拿到 orders，打完收工
}

第 2 步：手动跑一遍，理解”吐出去 → 等结果 → 喂回来”

const gen = myFlow();

const step1 = gen.next();     // 跑到 ①，吐出 fetchUser('Bob')
// step1 = { value: Promise<user>, done: false }

step1.value.then(user => {
  const step2 = gen.next(user); // 把 user 喂回去，跑到 ②，吐出 fetchOrders
  // step2 = { value: Promise<orders>, done: false }

  step2.value.then(orders => {
    gen.next(orders);          // 把 orders 喂回去，跑到 ③，done: true
  });
});

手动一个个写 .then() 太蠢了——写个循环让它自动跑：

第 3 步：写个几行的小马达，自动驱动

function run(genFn) {
  const gen = genFn();
  function next(val) {
    const { value, done } = gen.next(val);
    if (done) return;                // 跑完了
    value.then(next);                  // 等 Promise 完成，拿结果继续跑
  }
  next();  // 点火
}

// 用起来——异步代码写成一行行，清晰直白
run(function* () {
  const user   = yield fetchUser('Bob');
  const orders = yield fetchOrders(user.id);
  console.log(orders);
});
🎯 一句话总结：生成器遇到 yield 就暂停，run() 在外面等着 Promise 完成，完事了用 next(结果) 把值塞回去、让它继续跑。坏消息：自己要写 run()。好消息：ES2017 把 function* → async function、yield → await、run() → 引擎内置，所以你直接用 async/await 就行了，不用关心里面的马达怎么转。
🧠 面试只需要记住：async/await = generator + 自动执行器。引擎把 yield 换成 await，帮你内置了上面那个 run()。

1.5 Streams（Node.js 最核心的抽象）

Stream 是 Node.js 处理数据的核心抽象——把数据看作水流，不一次性加载到内存，而是"流到哪处理到哪"。

1.5·附：五种 Stream 类型速览

类型角色核心方法一句话
Readable 数据源（生产者） push(chunk) / push(null) 往外吐数据
Writable 数据目的地（消费者） write(chunk) / end() 往里灌数据
Transform 既是 Readable 又是 Writable _transform(chunk, enc, cb) 边读边写边转换（如 gzip 压缩）
Duplex 独立的 Readable + Writable _read() + _write() 双向管道（如 TCP socket）
PassThrough Transform 的空实现 不做任何转换，透传 用于管道观察/打日志/测试
📌 Transform vs Duplex 的区别：Transform 的输入和输出有因果关联（你写 A → 它读出 A'），Duplex 的读和写互不相干（你写不写它都自己读）。新手最容易搞混。

1.5·附：背压机制（backpressure）——为什么 stream 不会爆内存

场景：磁盘读速 500MB/s，网络写速 10MB/s。如果不做控制，数据会在内存里堆成山。

背压原理：下游写不进去时，上游暂停读取。

const rs = fs.createReadStream('/big-file.bin');
const ws = fs.createWriteStream('/copy.bin');

// ws.write() 的返回值是"还能不能继续写"
// 生产可写流 → writable 内部有个缓冲
// 缓冲满了 → ws.write() 返回 false → 应该暂停上游
// 缓冲排空 → drain 事件 → 可以继续

rs.on('data', chunk => {
  const canWrite = ws.write(chunk);
  if (!canWrite) {
    rs.pause();               // 下游写不动了，上游停
  }
});
ws.on('drain', () => {
  rs.resume();                // 下游消化完了，上游继续
});

1.5·附：手写一个组合的 Transform stream

import { Transform } from 'stream';
import { pipeline } from 'stream/promises';
import fs from 'fs';

// 一个实用的 Transform：给每行加行号 + 结尾输出统计
const lineNumberTransform = new Transform({

  // ① construct：异步初始化（在这里设置实例状态）
  construct(callback) {
    this.lineNumber = 0;
    this.remainder = '';          // 上次没拼完的半行，跨 chunk 缓冲区
    callback();
  },

  // ② transform：每来一块数据触发一次（唯一必须实现的方法）
  transform(chunk, encoding, callback) {
    const text = this.remainder + chunk.toString();
    const lines = text.split('\n');
    this.remainder = lines.pop();     // 最后一段是不完整的行，留到下次

    for (const line of lines) {
      this.lineNumber++;
      this.push(`[${this.lineNumber}] ${line}\n`);
    }
    callback();                        // 通知"这一块处理完了"
  },

  // ③ flush：仅在上游显式调用 end() 后触发，不会被空闲超时等任何其他条件触发
  // 如果上游一直 write 但不 end，flush 永远不会执行
  flush(callback) {
    if (this.remainder) {               // 处理最后可能残留的半行
      this.lineNumber++;
      this.push(`[${this.lineNumber}] ${this.remainder}\n`);
    }
    this.push(`\n--- 共 ${this.lineNumber} 行 ---\n`);
    callback();
  },

  // ④ destroy：流被销毁时调用，释放自定义资源
  destroy(err, callback) {
    this.remainder = null;            // 清空引用，帮助 GC
    callback(err);                   // 必须透传 err，否则 'close' 事件不会触发
  }
});

// 事件监听：观察生命周期（生产环境通常只绑 error）
lineNumberTransform.on('end',    () => console.log('[事件] 可读侧结束'));
lineNumberTransform.on('finish', () => console.log('[事件] 可写侧结束'));
lineNumberTransform.on('close',  () => console.log('[事件] 流已关闭'));

// pipeline 自动处理背压 + 错误传播
await pipeline(
  fs.createReadStream('input.txt'),
  lineNumberTransform,
  fs.createWriteStream('output.txt')
);
console.log('管道完成');

1.5·附：什么时候用 stream，什么时候不用

场景用原因
文件 < 100MB fs.readFile 一次读简单省事，内存够用
文件 > 1GB fs.createReadStream stream 是唯一选择，否则爆内存
大文件转换（压缩/解压/加密） pipeline(read → transform → write) 逐块处理，不存全量
HTTP 请求/响应体（未知大小） stream（req/res 本身就是 stream） 数据可能很大，不能全 load
数据库查询结果（百万行） 游标 + stream cursor 支持逐条流式读取

你写的每一个文件，Node 在执行前会包上一层函数：

// 你写的 foo.js：
const x = 1;
module.exports = { x };

// Node 编译时变成：

  const x = 1;

参数是什么常见用途
exports module.exports 的快捷引用 exports.foo = 1 ✅
exports = {foo: 1} ❌ 切断了引用
require 加载其他模块的函数 require('./bar')
module 当前模块对象（含 exports、id、parent） module.exports = ...

文件扩展名是什么系统需要 "type": "module" 吗

二、Node.js 内核机制（分水岭）

2.1 ：libuv 线程池与 I/O 完成端口的协作

为什么需要线程池？——两类 I/O 的根本区别

操作系统提供的异步 API不是全能的。把 I/O 分两路看：

网络 I/O（socket、pipe）文件 I/O + 计算密集操作
OS 支持 ✅ 内核原生异步
epoll(Linux) / kqueue(macOS) / IOCP(Windows)		 ⚠️ POSIX 规范里文件 I/O 是同步阻塞的
read()/write() 不提供异步版本（Linux AIO 有坑）
libuv 怎么做 直接对接内核通知机制，不占线程 把阻塞操作扔进线程池，用线程模拟异步
线程在后台干活，干完通知事件循环
消费线程吗 ❌ 0 个线程（纯内核态） ✅ 每个操作占 1 个线程（默认池大小 4）
📌 一句话区分：网络 I/O 走事件通知（内核告诉你"数据到了"），文件 I/O 走线程池（线程帮你读完文件再通知你）。前者是真正的异步，后者是"用线程伪装的异步"。

线程池怎么工作的

🔵 JavaScript 层：fs.readFile('/data.txt', cb) → 把请求交给 libuv
🟡 libuv 层：(1) 网络 I/O → 注册到 epoll，直接返回  |  (2) 文件 I/O → 扔进 Work Queue
🟣 线程池：4 个工作线程轮询 Work Queue → 其中一个拿到任务 → 执行阻塞 read()
🔴 完成通知：线程执行完 → 把 cb 推入事件循环的 pending 队列 → 下一轮 poll 阶段执行 cb

关键：JavaScript 主线程自始至终没离开过事件循环——它只是把活丢给 libuv，然后继续处理其他事情。阻塞发生在线程池的线程里，不影响主线程。

2.1·附：进线程池 vs 不进——完整的操作清单

分类API / 模块走线程池？原因
文件系统 fs.readFile / fs.writeFile ✅ 进 POSIX 文件 I/O 没有可靠的异步 API
fs.readdir / fs.stat ✅ 进 目录和元数据操作同属文件系统
fs.open / fs.close ✅ 进 同上
fs.createReadStream ✅ 进 底层读操作仍依赖线程池
密码学 crypto.pbkdf2 / crypto.scrypt ✅ 进 CPU 密集，不占线程池会饿死事件循环

DNS dns.lookup() ✅ 进 调用 getaddrinfo()，这是阻塞系统调用

⚠️ 常见坑：dns.lookup() 走线程池（调 getaddrinfo），dns.resolve() 不走（用 c-ares 库自己发 DNS 包）。大量并发 DNS 查询请用 dns.resolve* 系列，避免占满线程池。

2.1·附：UV_THREADPOOL_SIZE 调优——什么时候动它

默认值

线程池默认 4 个线程。意味着同时最多只能有 4 个线程池操作（fs/crypto/zlib/dns.lookup）在并行执行。第 5 个操作排队等待。

判断是否需要调大

不是越大越好。线程多了 → 上下文切换开销增大 → 反而可能更慢。只在以下信号出现时考虑调大：

信号诊断方法推荐值
大量并发文件读写 高并发下 fs 的 p99 延迟明显高于 p50 8 ~ 16
密码哈希成为瓶颈 crypto.pbkdf2 调用堆积，CPU 却用不满 等于 CPU 核数
4 个以上 CPU 核，但 CPU 利用率低 多核机器上 I/O 吞吐上不去 CPU 核数 ~ 核数×2
zlib 大批量压缩/解压 压缩吞吐量瓶颈 8 ~ CPU 核数

怎么设

// 方式一：启动时环境变量（推荐）
// $ UV_THREADPOOL_SIZE=8 node server.js

// 方式二：代码里设（必须在所有异步操作之前！）
process.env.UV_THREADPOOL_SIZE = 8;
const fs = require('fs');
// 如果放在 require('fs') 之后，libuv 已经初始化完毕，设了也没用
⚠️ process.env.UV_THREADPOOL_SIZE 必须写在所有 require 之前！libuv 在第一次异步操作时初始化线程池，之后再改这个值无效。

快速验证：用 crypto 榨干线程池

const start = Date.now();
// 同时发起 8 个 pbkdf2（比线程池大 1 倍）
for (let i = 0; i < 8; i++) {
  crypto.pbkdf2('password', 'salt', 100000, 512, 'sha512', () => {
    console.log(`第 ${i + 1} 个完成`, Date.now() - start, 'ms');
  });
}
// UV_THREADPOOL_SIZE=4：分两批，每批 4 个同时跑，第一批 ~1.5s，第二批 ~3s
// UV_THREADPOOL_SIZE=8：8 个同时跑，全部 ~1.5s

2.2 V8 堆结构

  • 新生代 (Scavenge) / 老生代 (Mark-Sweep & Mark-Compact)

2.2·附：V8 堆结构与 GC 详解

🏗️ V8 堆的布局：一张地图

V8 的堆不是一整块，而是分成了几个具有不同用途的区域。每个区域用不同的 GC 策略管理：

空间放什么GC 策略大小
New Space（新生代） 刚创建、生命周期短的对象 Scavenge（复制算法） 1~8 MB（64 位系统通常 16MB 弱分代）
Old Space（老生代） 活过多次 GC 的对象 Mark-Sweep + Mark-Compact 可达 1.4GB（64 位）
Code Space JIT 编译后的机器码 同老生代 动态
Large Object Space 大于其他空间 page 大小的对象 独立管理，不移动 按需
Map Space 对象的 Hidden Class（Map） 同老生代 动态

🔑 核心设计思想：分代假说（Generational Hypothesis）
绝大多数对象"朝生暮死"——创建之后很快就变成垃圾。只有少数对象活得很久。所以新生代用快但不耐大的 Scavenge，老生代用慢但能扛大的 Mark-Sweep。

🧹 新生代：Scavenge 复制算法

核心机制：Semispace（双空间）

新生代被切成两半：From 空间（活跃对象所在）和 To 空间（空闲）。GC 时不是"标记-清除"，而是把活对象拷贝到 To 空间，然后整块 From 空间直接作废。

From 空间（有对象） → GC 时把活对象拷到 To → From 整块清空
下一次 GC：From ↔ To 角色互换（To 变 From，From 变 To） 
// Scavenge 的过程（简化）
// 1. 从根集合（全局变量、栈变量）出发，遍历所有引用
// 2. 每找到一个活对象，拷到 To 空间
// 3. 拷完更新指针，让它指向新地址
// 4. 复制完所有活对象后，整块 From 空间被标记为空闲
// 5. From 和 To 身份互换，下次 GC 再来
✅ Scavenge 的优点：
• 只处理活对象，不关心死对象——新生代里大多数是死的，所以很快
• 天然无碎片——复制时对象紧密排列，To 空间永远连续
• O(存活对象数) 而不是 O(堆大小)
⚠️ Scavenge 的代价：
• 永远有一半空间空着（To 空间闲置），内存利用率只有 50%
• 这就是为什么新生代不能太大——浪费不起

对象晋升（Promotion）：从新生代到老生代

一个对象在新生代里不是永久的。满足以下条件之一就会被迁移到老生代：

  1. 活过两次 Scavenge：已经经历了一次 GC，这次又被拷到 To 空间 → 直接晋升，不拷到 To 了
  2. To 空间塞不下：存活对象超过了 To 空间的 25% → 直接晋升
// 对象的一生（简化）
const obj = {};          // ① 出生在新生代 From 空间
// ... 第一次 Scavenge ...  // ② 还活着 → 拷到 To 空间
// ... 第二次 Scavenge ...  // ③ 又活着 → 晋升到老生代
// ... Mark-Sweep ...      // ④ 在老生代被管理，直到没人引用

🧹 老生代：Mark-Sweep & Mark-Compact

老生代的对象大、存活久、数量多。Scavenge 那套"拷贝活对象"不适用了——拷贝成本太高。所以换了两套算法：

Mark-Sweep（标记-清除）——日常主力

分两个阶段：

① Mark（标记）：从根开始遍历所有可达对象，给它们打上"活的"标记
② Sweep（清除）：扫描整个老生代，没标记的空间回收掉 
// Mark-Sweep 的碎片问题
// 回收前：
//   [A][ ][B][ ][C]   ← A、B、C 是活对象，之间有空洞
// 回收后：
//   [A][ ][B][ ][C]   ← 空洞还在！新对象要找合适大小的洞塞进去
// 久而久之，老生代布满空洞 → 这就是"内存碎片"

Mark-Compact（标记-整理）——碎片太多时出场的救火队

当 Mark-Sweep 之后内存碎片化严重，以至于分配一个大对象找不到连续空间时，触发 Mark-Compact：

// Mark-Compact：把所有活对象推到一端
// 整理前：
//   [A][ ][B][ ][ ][C]   ← 活对象散落，空洞遍布
// 整理后：
//   [A][B][C][ ][ ][ ]   ← 活对象紧密排列，空闲空间连续
⚠️ Mark-Compact 慢。因为要移动对象 + 更新所有指向它的指针。所以 V8 只在必要的时候才做，日常还是 Mark-Sweep。

⚡ Orinoco：让 GC 不再"Stop-The-World"

传统 GC 最致命的问题：GC 工作的时候，JS 代码全部暂停（Stop-The-World）。一个老生代 GC 可能停几十到上百毫秒，对服务端就是请求延迟的毛刺。

V8 的 Orinoco 项目把 GC 工作拆分、并行化、甚至搬到后台线程：

技术含义哪个阶段用
Parallel（并行） 标记或整理时，多个辅助线程一起干。
主线程仍然暂停，但暂停时间大幅缩短。		 新生代 Scavenge
老生代 Sweep
Incremental（增量） 把标记工作拆成多个小片，和 JS 代码交替执行。
每片只停几毫秒，用户感知不到。		 老生代 Mark
Concurrent（并发） 标记工作完全在后台线程进行，主线程不暂停。
JS 代码和 GC 同时跑。		 老生代 Mark
老生代 Sweep（部分）
🧠 一句话总结 Orinoco：Mark 阶段做增量 + 并发，Sweep/Compact 阶段做并行。目标是让每次 GC 暂停短到用户感知不到（目标 <1ms）。

📊 V8 GC 的实际表现

GC 类型触发场景暂停时间频率
Scavenge（新生代） New Space 满了 < 1ms 频繁（每秒多次）
Incremental Mark 老生代达到阈值 每片 0.5~2ms 由老生代增长速度决定
Full Mark-Compact 碎片化严重 或 增量标记追不上分配速度 可能数十到上百毫秒 罕见，应尽量避免
🎯 对你写代码意味着什么：
① 别频繁创建大临时对象——它可能跳过新生代直接进老生代，触发 Full GC
② 老对象别引用新对象——这会增加写屏障（write barrier）开销
③ GC 不是你的敌人——正常业务代码 V8 处理得很好，别过度优化

2.3·附：堆快照分析实操（Chrome DevTools 可复现）

🎯 一句话讲清堆快照是什么

堆快照（Heap Snapshot）就是 JS 堆在某一时刻的完整照片——拍下了当时所有活着的对象、它们之间的引用关系、以及每个对象占了多少内存。你拿着这张照片可以找到"谁最大"、"谁不必要地活着"、"谁在两次快照之间冒出来了"。

三个核心指标：
Shallow Size对象自身占用的内存（不算它引用的东西）
Retained Size对象自身 + 它独占引用的所有子对象的内存总和。删掉这个对象能释放多少内存——这是找泄漏的核心指标
Distance从 GC 根（root）到这个对象最短要经过几层引用

2.4 内存安全实践

  • 能写出不泄漏的闭包、不爆炸的 Buffer 使用、正确的 Stream 消费

2.4·附：内存安全实践详解

💡 心法：Node.js 的内存问题分三类——闭包抓太多（引用未释放）、Buffer 没上限（量太大撑爆）、Stream 没消费（背压失效导致积压）。本质都是"该放手时没放手"。GC 只能回收没人引用的对象，你的代码只要还抓着引用，GC 就无能为力。

一、闭包泄漏：不是你写了闭包就泄漏，是闭包抓了不该抓的

闭包的本质是函数携带了它的词法环境。每创建一个函数，它就引用着外层作用域的所有变量——哪怕你只用了一个。

❌ 经典泄漏 1：回调里抓了大对象

function processUpload(fileBuffer) {
  const metadata = extractMetadata(fileBuffer);  // 只需要这一小段

  // ❌ 闭包引用了整个 fileBuffer（可能几百 MB）
  const cleanup = () => {
    console.log(`处理完成: ${metadata.name}`);
    // 这里没用到 fileBuffer,但闭包把整个外层作用域都抓了
  };

  scheduleCleanup(cleanup);  // cleanup 活着 → fileBuffer 不能被 GC
}

✅ 修复：只传需要的数据

function processUpload(fileBuffer) {
  const metadata = extractMetadata(fileBuffer);
  const name = metadata.name;  // 只取需要的字段

  // ✅ 闭包只抓了 name（一个字符串），fileBuffer 可以被 GC
  const cleanup = () => console.log(`处理完成: ${name}`);
  scheduleCleanup(cleanup);
}

❌ 经典泄漏 2：事件监听器没取消

// ❌ 每次请求注册一个监听器，从不移除
server.on('request', (req) => {
  const heavyData = loadHeavyData();  // 很大的数据

  req.on('end', () => {
    // end 事件的闭包引用了 heavyData
    // 即使请求处理完了,这个闭包还挂在 req 上（req 可能被连接池复用）
    logRequest(heavyData);
  });
});

✅ 修复：用完就解绑

server.on('request', (req) => {
  const heavyData = loadHeavyData();

  const onEnd = () => {
    logRequest(heavyData);
    req.removeListener('end', onEnd);  // ✅ 用完移除
  };
  req.on('end', onEnd);
});

❌ 经典泄漏 3：定时器/Interval 持有闭包

let sessions = new Map();

function createSession(userData) {
  const id = generateId();
  sessions.set(id, userData);

  // ❌ 定时器持有的闭包引用了 userData
  // userData 被 sessions 和 setInterval 双重持有
  // 即使从 sessions 里 delete 了,定时器还抓着 → 泄漏
  const timer = setInterval(() => {
    refreshToken(userData.token);
  }, 60000);

  return id;
}

✅ 修复：生命周期绑定，用 WeakRef 或主动清理

const timers = new Map();  // sessionId → timer

function destroySession(id) {
  clearInterval(timers.get(id));  // ✅ 先清定时器
  timers.delete(id);
  sessions.delete(id);           // 再清数据
}

🧪 怎么在堆快照里验证？

  1. 搜索你的变量名（如 fileBuffer），看 Retained Size
  2. 在 Retainers 面板看引用链——如果看到 system / Context → 闭包持有 → 全局对象，就是泄漏
  3. 对比两次快照：如果同名的闭包对象 Delta 一直在涨，确认为泄漏

二、Buffer 爆炸：不是 Buffer 本身大，是"积压"把它撑大了

Buffer 本身没有泄漏机制——Buffer.alloc() 之后只要没引用就会被 GC。真正的问题是并发未受控地创建大量 Buffer，或者请求体没有上限导致单次分配过大。

❌ 场景 1：没有限制请求体大小

// ❌ 攻击者发一个 10GB 的 POST body → 服务器 OOM
const server = http.createServer(async (req, res) => {
  const chunks = [];
  for await (const chunk of req) {
    chunks.push(chunk);  // 所有 chunk 都囤在内存里，没有上限
  }
  const body = Buffer.concat(chunks);
  await processBody(body);
  res.end('ok');
});

✅ 修复：加大小限制 + 流式处理

const MAX_BODY = 10 * 1024 * 1024;  // 10MB 上限

const server = http.createServer(async (req, res) => {
  const contentLength = parseInt(req.headers['content-length'], 10);
  if (contentLength > MAX_BODY) {
    res.statusCode = 413;
    res.end('Payload Too Large');
    return;
  }

  const chunks = [];
  let received = 0;
  for await (const chunk of req) {
    received += chunk.length;
    if (received > MAX_BODY) {         // 双重检查（Content-Length 可能伪造）
      res.statusCode = 413; res.end(); return;
    }
    chunks.push(chunk);
  }
  const body = Buffer.concat(chunks);
  await processBody(body);
  res.end('ok');
});

❌ 场景 2：并发创建 Buffer 无上限

// ❌ 1000 个请求同时进来 → 1000 × 10MB = 10GB → OOM
app.get('/report', async (ctx) => {
  const report = generateLargeReport();  // 生成 10MB 的报告 Buffer
  ctx.body = report;
});

✅ 修复：并发限流 + 流式返回

const { Readable } = require('stream');

app.get('/report', async (ctx) => {
  // ✅ 流式返回——不需要一次性创建整个 Buffer
  ctx.type = 'text/csv';
  ctx.body = new Readable({
    read() {
      const row = generateNextRow();  // 逐行生成
      if (row) this.push(row);
      else this.push(null);       // 结束
    }
  });
});

❌ 场景 3：循环里频繁创建 Buffer

// ❌ 每次循环 new 一个 Buffer → 频繁 GC → 性能抖动
for (let i = 0; i < 100000; i++) {
  const buf = Buffer.alloc(4096);  // 每次都分配新内存
  processChunk(buf);
}

✅ 修复：Buffer 池复用

class BufferPool {
  pool = [];
  acquire(size) {
    const buf = this.pool.pop();
    if (buf && buf.length >= size) return buf;
    return Buffer.allocUnsafe(size);  // allocUnsafe 不初始化,比 alloc 快
  }
  release(buf) { this.pool.push(buf); }
}

const pool = new BufferPool();
for (let i = 0; i < 100000; i++) {
  const buf = pool.acquire(4096);
  processChunk(buf);
  pool.release(buf);  // ✅ 归还给池,下次复用
}
⚠️ Buffer.allocUnsafe 不初始化内存，可能包含旧数据（安全风险）。只用在内部分配、立刻覆盖的场景。对外暴露的数据用 Buffer.alloc（零初始化）。

三、Stream 背压：忽略了背压是内存泄漏的隐藏炸弹

Stream 的核心设计是背压（Backpressure）：消费端处理速度 < 生产端写入速度时，Stream 会自动暂停生产端，防止内存中积压数据。

但背压不是自动生效的——必须正确返回 write() 的返回值并监听 drain 事件。

❌ 经典错误：忽略 write() 返回值

const fs = require('fs');

// ❌ 不管 write() 返回 false → 背压失效
// 数据在内存中无限积压（Writable 的内部缓冲）
// 读取 1GB 文件时 → 整个文件内容都进了内存 → OOM
const readStream = fs.createReadStream('huge-file.bin');
const writeStream = fs.createWriteStream('copy.bin');

readStream.on('data', (chunk) => {
  writeStream.write(chunk);  // ← 返回 false 时表示"我写不下了,暂停"
  // 但代码没管 → 数据继续涌入内存
});

✅ 正确写法 1：用 pipe()（最简单,内置背压处理）

// ✅ pipe() 自动处理背压——99% 的场景用它就够了
fs.createReadStream('huge-file.bin').pipe(fs.createWriteStream('copy.bin'));

✅ 正确写法 2：手动监听 drain（需要自定义处理逻辑时）

const readStream = fs.createReadStream('huge-file.bin');
const writeStream = fs.createWriteStream('copy.bin');

readStream.on('data', (chunk) => {
  const canContinue = writeStream.write(chunk);
  if (!canContinue) {
    // ✅ 写不下了 → 暂停读取
    readStream.pause();

    // ✅ 等写端排空了 → 恢复读取
    writeStream.once('drain', () => {
      readStream.resume();
    });
  }
});

❌ Transform Stream 的背压陷阱

Transform Stream 有内部缓冲。如果 _transform 是异步的但没正确处理背压，缓冲里的数据会越积越多：

const { Transform } = require('stream');

// ❌ 异步 transform 但没有处理背压
const slowTransform = new Transform({
  async transform(chunk, encoding, callback) {
    const result = await slowDbQuery(chunk);  // 每次 200ms
    this.push(result);
    callback();
  }
});

// 上游 1000 chunk/s 进来 → transform 200ms/chunk → 内部缓冲爆炸
fastSource.pipe(slowTransform).pipe(destination);

✅ 修复：控制并发 + 正确使用 callback

const slowTransform = new Transform({
  highWaterMark: 16,  // ✅ 限制内部缓冲大小（默认 16 个对象）
  async transform(chunk, encoding, callback) {
    try {
      const result = await slowDbQuery(chunk);
      this.push(result);
      callback();  // ✅ 处理完才 callback,告诉上游可以再发
    } catch (err) {
      callback(err); // ✅ 错误也要传,否则流会卡住
    }
  }
});

Reader/Consumer 速度不匹配时的内存影响

上游(生产)                    流内部缓冲                         下游(消费)
10 MB/s 写入  ──→  [============  积压缓冲  ============]  ──→  1 MB/s 读取

如果没背压:
  缓冲里的数据线性增长 → 内存无限膨胀 → OOM

如果有背压(drain):
  缓冲满时 write() 返回 false → 上游暂停 → 缓冲被消费 → drain 事件 → 上游恢复

四、排查心法：内存问题的万能公式

  1. 确认是泄漏还是正常使用：生产环境 RSS 持续增长且不回落 → 泄漏。RSS 波动但 GC 后回落 → 正常。
  2. 取两个堆快照：一个是基线，一个是内存上涨后。间隔至少让内存涨一倍。
  3. Comparison 视图查 Delta：看 Constructor 的 `# New` / `# Delta` / `Size Delta`，增长最大的展开。
  4. Retainers 面板追引用链：选中可疑对象 → 看 Retainers → 追溯到 GC Root → 找到根源代码。
  5. 修好后验证：同样条件再取快照 → Delta 应为零或下降。

最常见的三类根源：全局变量/缓存无上限、事件监听器未解绑、闭包持有大对象不放。

✅ 一句话总结：GC 能回收"没人引用的对象"。你的任务不是帮 GC，而是确保该放手时引用链断了——闭包缩小抓取范围、Buffer 加容量上限、Stream 正确处理背压。

});
// 你以为大小 7，实际是 12，b 的 offset 错误

// ✅ 正确：理解 alignment
const GOOD = koffi.struct('GOOD', {
    a: 'uint8',   // offset 0
    b: 'uint32',  // offset 4 ← 对齐到 4 字节边界
    c: 'uint16',  // offset 8
});
准则：要么用工具自动生成 binding，要么对着 C 的 sizeof 和 offsetof 结果逐字段验证。错一个字节 = UB（通常 segfault 或静默数据损坏）。

Layer 3：N-API / node-addon-api 写 C++ 插件

当你需要的不是 1-2 个 API 调用，而是一个完整的 C++ 库集成（OpenCV、硬件驱动 SDK）。

示例：调 GetDiskFreeSpace

// 用 node-addon-api (C++ 封装层)
#include <napi.h>
#include <windows.h>

Napi::Value GetDiskFreeSpace(const Napi::CallbackInfo& info) {
    Napi::Env env = info.Env();

    DWORD sectorsPerCluster, bytesPerSector, freeClusters, totalClusters;
    BOOL result = GetDiskFreeSpaceW(
        L"C:\\",            // 实际应从参数获取
        &sectorsPerCluster,
        &bytesPerSector,
        &freeClusters,
        &totalClusters
    );

    if (!result) {
        DWORD err = GetLastError();  // ← 必须立即取！
        Napi::Error::New(env, "GetDiskFreeSpace failed: " + std::to_string(err))
            .ThrowAsJavaScriptException();
        return env.Undefined();
    }

    // 用 ULONGLONG 避免乘法溢出
    ULONGLONG freeBytes = static_cast<ULONGLONG>(freeClusters)
                        * sectorsPerCluster * bytesPerSector;

    return Napi::Number::New(env, static_cast<double>(freeBytes));
}

3.1 你需要的 C++ 基础

  • RAII、智能指针、异常安全
  • 多线程与互斥锁（uv_queue_work 把重活放到 libuv 线程池）
  • 内存管理：new/delete vs malloc/free 不能混用

3.2 V8 嵌入模型

  • Handle / HandleScope：GC 根管理，HandleScope 溢出 = 内存泄漏
  • Isolate / Context：理解 V8 隔离模型，worker_threads 每个线程有独立 Isolate
  • Persistent handle：跨 GC 保持引用的唯一方式，必须手动 Reset()
  • FunctionTemplate / ObjectTemplate：原型链映射
⚠️ Native Addon 第一大坑：线程安全
C++ addon 的回调可能来自多个线程：
  • Node.js Event Loop 线程 → ✅ 安全，可以调 JS
  • 你自己的 C++ 线程 → ❌ 不安全！必须用 Napi::ThreadSafeFunction 桥接到 JS
  • libuv 线程池 → 通过 uv_async_send 通知主线程
错误的线程访问 V8 对象 → 随机崩溃。

Node.js 本身不直接 syscall，一切通过 libuv → 系统调用。

你在 JavaScript 的堆上跑，但在 C 的栈上死。
每次 FFI 调用都是一次跨越 V8 沙箱边界的跳跃。
能安全来回的，是那些精确理解了两边内存模型的人。

把 koffi 的 struct 定义写得和 C 的 sizeof() 一模一样，把 GetLastError() 和 errno 在正确时机取出，把每个 HANDLE/FD 的生命周期管理清楚——这些就是区分"能用"和"精通"的地方。
