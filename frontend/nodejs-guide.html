⚡ 微任务插队机制（面试必考）

核心规则：每个宏任务阶段执行完毕后、切换到下一阶段之前，Node 会先清空微任务队列。顺序严格为：
① 清空 nextTick 队列（全部）→ ② 清空 Promise 微任务队列（全部）

🧪 经典面试题：逐行执行跟踪

// 完整执行顺序是什么？
console.log('1. sync start');

setTimeout(() => console.log('2. setTimeout'), 0);

setImmediate(() => console.log('3. setImmediate'));

Promise.resolve()
  .then(() => console.log('4. Promise.then 1'))
  .then(() => console.log('5. Promise.then 2'));

process.nextTick(() => console.log('6. nextTick 1'));
process.nextTick(() => {
  console.log('7. nextTick 2');
  process.nextTick(() => console.log('8. nextTick 3 (嵌套)'));
});

console.log('9. sync end');
📋 执行顺序 + 详解：
=== 同步代码（调用栈） === 1. sync start ← setTimeout → timers / setImmediate → check 9. sync end ← Promise / nextTick → 微任务队列 === 同步栈清空 → 微任务检查 === 6. nextTick 1 ← nextTick 优先于 Promise！ 7. nextTick 2 ← 执行过程中追加了 8 8. nextTick 3 (嵌套) ← nextTick 队列必须全部清空 4. Promise.then 1 ← nextTick 清空后 → Promise 5. Promise.then 2 ← .then 链入队 === 微任务清空 → Event Loop === → ① timers：setTimeout 到期 2. setTimeout → ②③④ 流转... → ⑤ check 3. setImmediate

⭐ 关键结论（背下来）

  1. nextTick > Promise.then：每次微任务检查，先清空 nextTick 再清空 Promise
  2. setImmediate > setTimeout(0)：check 在 poll 之后立刻执行，setTimeout 等下轮 timers
    例外：在 I/O 回调内部，setImmediate 永远先于 setTimeout——因为 I/O 回调在 poll 执行，poll 之后就是 check
  3. nextTick 嵌套不降级：回调里再调 nextTick，追加到当前队列，本轮清空（可能"饿死"事件循环）
  4. Promise.then 链式调用：每个 .then 是一次新的微任务入队，级联执行

第二部分：从 Node.js 调用操作系统底层接口

先明确目标：四个层级

Layer 1   exec / spawn 调系统命令   →   懂 OS 命令即可
Layer 2   FFI 调动态库 (.dll / .so)   →   懂 C ABI、类型映射
Layer 3   N-API / node-addon-api 写插件   →   懂 C/C++、V8 嵌入
Layer 4   直接 syscall（极少）  →   懂内核接口、架构差异

Layer 1：child_process / exec

通用 覆盖 70% 场景

// Windows
execSync('powershell Get-NetAdapter')
execSync('wmic cpu get name')

// Linux
execSync('cat /proc/cpuinfo')
execSync('ip link show')

需要知道：

  • 目标平台的常用命令行工具
  • 输出解析（不同 locale / 版本输出格式不同，这是个常踩的坑）
  • 参数注入防范（永远不要拼接用户输入到 shell 命令）
  • Win cmd.exe 默认编码是 GBK → 需要 {encoding: 'buffer'} 或 chcp 65001
✅ 不需要 OS 内核知识。

Layer 2：FFI 调用系统动态库（推荐优先学）

通过 ffi-napi / koffi / bun:ffi 直接调用 .dll / .so。

示例：调 Kernel32 获取系统信息

const koffi = require('koffi');
const kernel32 = koffi.load('kernel32.dll');

const SYSTEM_INFO = koffi.struct('SYSTEM_INFO', {
    wProcessorArchitecture: 'uint16',
    wReserved:              'uint16',
    dwPageSize:             'uint32',
    lpMinimumApplicationAddress: 'void*',
    lpMaximumApplicationAddress: 'void*',
    dwActiveProcessorMask:  'uint64',
    dwNumberOfProcessors:   'uint32',
});

kernel32.stdcall('GetSystemInfo', SYSTEM_INFO, [koffi.out(SYSTEM_INFO)]);

2.2 调用约定 (Calling Convention)

平台约定规则
Win stdcall Win32 API 标准，被调用者清栈
Win fastcall x64 统一约定：前 4 个整数参数 → rcx, rdx, r8, r9
Linux System V AMD64 x64 统一约定：前 6 个整数参数 → rdi, rsi, rdx, rcx, r8, r9
⚠️ 致命坑：调用约定不匹配 → 栈损坏 → 进程崩溃。这是 FFI 调用中最常见的致命错误。

2.3 结构体对齐 (Alignment / Padding)

// ❌ 错误：按直觉写
const BAD = koffi.struct('BAD', {
    a: 'uint8',   // offset 0
    b: 'uint32',  // offset 1 ← 错！实际 C 编译 offset 是 4
    c: 'uint16',  // offset 5 ← 错！实际是 8
});
// 你以为大小 7，实际是 12，b 的 offset 错误

// ✅ 正确：理解 alignment
const GOOD = koffi.struct('GOOD', {
    a: 'uint8',   // offset 0
    b: 'uint32',  // offset 4 ← 对齐到 4 字节边界
    c: 'uint16',  // offset 8
});
准则：要么用工具自动生成 binding，要么对着 C 的 sizeof 和 offsetof 结果逐字段验证。错一个字节 = UB（通常 segfault 或静默数据损坏）。

2.5 指针与内存生命周期

const buf = Buffer.alloc(256);
NativeFunction(buf);
// buf 在下次 GC 前不会被回收吗？← 不确定！
  • 规则 1：确保 JS 变量在 C 使用期间保持引用
  • 规则 2：C 侧如果要保存指针，必须在 C 堆上分配，不能指回 V8 堆。V8 的 compacting GC 会移动对象
  • 规则 3：Buffer 的底层 ArrayBuffer 通常固定在 V8 外，相对安全，但不是绝对的

Layer 3：N-API / node-addon-api 写 C++ 插件

当你需要的不是 1-2 个 API 调用，而是一个完整的 C++ 库集成（OpenCV、硬件驱动 SDK）。

示例：调 GetDiskFreeSpace

// 用 node-addon-api (C++ 封装层)
#include <napi.h>
#include <windows.h>

Napi::Value GetDiskFreeSpace(const Napi::CallbackInfo& info) {
    Napi::Env env = info.Env();

    DWORD sectorsPerCluster, bytesPerSector, freeClusters, totalClusters;
    BOOL result = GetDiskFreeSpaceW(
        L"C:\\",            // 实际应从参数获取
        &sectorsPerCluster,
        &bytesPerSector,
        &freeClusters,
        &totalClusters
    );

    if (!result) {
        DWORD err = GetLastError();  // ← 必须立即取！
        Napi::Error::New(env, "GetDiskFreeSpace failed: " + std::to_string(err))
            .ThrowAsJavaScriptException();
        return env.Undefined();
    }

    // 用 ULONGLONG 避免乘法溢出
    ULONGLONG freeBytes = static_cast<ULONGLONG>(freeClusters)
                        * sectorsPerCluster * bytesPerSector;

    return Napi::Number::New(env, static_cast<double>(freeBytes));
}

3.1 你需要的 C++ 基础

  • RAII、智能指针、异常安全
  • 多线程与互斥锁（uv_queue_work 把重活放到 libuv 线程池）
  • 内存管理：new/delete vs malloc/free 不能混用

3.2 V8 嵌入模型

  • Handle / HandleScope：GC 根管理，HandleScope 溢出 = 内存泄漏
  • Isolate / Context：理解 V8 隔离模型，worker_threads 每个线程有独立 Isolate
  • Persistent handle：跨 GC 保持引用的唯一方式，必须手动 Reset()
  • FunctionTemplate / ObjectTemplate：原型链映射
⚠️ Native Addon 第一大坑：线程安全
C++ addon 的回调可能来自多个线程：
  • Node.js Event Loop 线程 → ✅ 安全，可以调 JS
  • 你自己的 C++ 线程 → ❌ 不安全！必须用 Napi::ThreadSafeFunction 桥接到 JS
  • libuv 线程池 → 通过 uv_async_send 通知主线程
错误的线程访问 V8 对象 → 随机崩溃。

3.3 Windows 特有知识

你需要懂的为什么
HANDLE 的引用计数CloseHandle 忘了 = 句柄泄漏 → 无法打开新文件/进程
ANSI vs Unicode API坚持用 W 后缀（CreateFileW），永远不用 A 后缀
GetLastError() 必须立即取任何后续 API 调用可能覆盖 error code
SEH 结构化异常__try/__except 与 C++ try/catch 是不同的两套机制；访问违规 (0xC0000005) 只能用 SEH
COM 模型WMI / DirectX / Shell：CoInitializeEx、IUnknown::Release、STA vs MTA
权限模型UAC、完整性级别、SeImpersonatePrivilege

3.4 Linux 特有知识

你需要懂的为什么
errno系统调用失败后必须检查，全局线程局部存储
内存映射 mmapMAP_SHARED vs MAP_PRIVATE vs MAP_ANONYMOUS vs MAP_POPULATE
文件描述符 FDFD 泄漏 = 文件描述符耗尽 = EMFILE
信号安全sigaction 的异步信号安全函数列表；信号处理器中能干的事极少
cgroup / namespace容器感知：/proc/self/cgroup、是否在容器内
CapabilitiesCAP_NET_RAW 等能力

Layer 4：直接 syscall（极少需要）

Node.js 本身不直接 syscall，一切通过 libuv → 系统调用。

const libc = koffi.load('libc.so.6');
libc.stdcall('syscall', 'long', ['long', '...']);

// gettid() - glibc 可能不暴露
const tid = libc.syscall(186); // SYS_gettid on x86-64
  • x86-64 syscall 编号（/usr/include/asm/unistd_64.h），不同架构不同编号
  • 寄存器传参约定（rax=syscall number, rdi/rsi/rdx/r10/r8/r9 依次传参）
  • 不同内核版本 syscall 行为差异
  • Win Windows 没有直接 syscall 接口，必须通过 ntdll.dll（NTDLL 是用户态到内核态的最终边界）

🧠 最关键的心法

你在 JavaScript 的堆上跑，但在 C 的栈上死。
